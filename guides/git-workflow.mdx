The recommended workflow with XcodeGen is to add `.xcodeproj` to your `.gitignore` and generate it on demand. This eliminates merge conflicts in project files and keeps your pull requests focused on actual code changes.

## Why gitignore your .xcodeproj

Xcode project files change frequently — adding a file, reordering a build phase, or changing a scheme setting all produce diffs in the `.pbxproj` file that are difficult to review and easy to conflict. By generating the project from a human-readable YAML spec:

- Pull requests only show changes to `project.yml` and source files
- Merge conflicts disappear — there is nothing to conflict
- Onboarding is simpler: clone, run `xcodegen generate`, open the project
- The project is always in sync with the files on disk

## Setting up .gitignore

Add the following to your `.gitignore`:

```
*.xcodeproj
!*.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved
```

The exception keeps the `Package.resolved` file in version control so everyone uses the same Swift package versions.

## Halfway approach

If removing `.xcodeproj` from git entirely is too large a change for your team, you can still use XcodeGen while keeping the project checked in. Run `xcodegen generate` after any change to `project.yml` and commit both files together. You lose the merge-conflict benefit but gain the consistency and readability of a spec-driven project.

## Regenerating after branch switches

When you check out a branch that adds or removes source files, your existing project will have stale references. You need to regenerate:

```bash
xcodegen generate
```

The `--use-cache` flag makes this fast — XcodeGen hashes the spec and all referenced files. If nothing changed, generation is skipped entirely:

```bash
xcodegen generate --use-cache
```

## Automating with git hooks

Git hooks let you regenerate the project automatically whenever you switch branches, merge, or rebase. Place hook scripts in `.git/hooks/`.

<Warning>
Hook scripts must be executable. After creating each file, run:

```bash
chmod +x .git/hooks/post-checkout
chmod +x .git/hooks/post-merge
chmod +x .git/hooks/post-rewrite
chmod +x .git/hooks/pre-commit
```
</Warning>

### post-checkout

Runs after `git checkout` and `git switch`. Regenerates the project if files changed on the new branch.

```bash
#!/bin/bash
xcodegen generate --use-cache
```

<Tip>
Hook path: `.git/hooks/post-checkout`
</Tip>

### post-merge

Runs after `git merge` and `git pull`. Regenerates when incoming changes affect the project.

```bash
#!/bin/bash
xcodegen generate --use-cache
```

<Tip>
Hook path: `.git/hooks/post-merge`
</Tip>

### post-rewrite

Runs after `git rebase` and `git commit --amend`. Regenerates when commits modify project-affecting files.

```bash
#!/bin/bash
xcodegen generate --use-cache
```

<Tip>
Hook path: `.git/hooks/post-rewrite`
</Tip>

### pre-commit

Runs before each commit. Updates the cache to reflect any local changes, including new files you added before staging. This ensures the cache stays accurate so that the post-checkout hooks on other machines don't skip regeneration when they should not.

```bash
#!/bin/bash
xcodegen cache
```

<Tip>
Hook path: `.git/hooks/pre-commit`
</Tip>

## Sharing hooks with your team

The `.git/hooks` directory is not tracked by git, so hooks won't be shared automatically. Common approaches:

<Tabs>
  <Tab title="Checked-in scripts">
    Commit your hooks to a directory in the repo (for example `scripts/hooks/`) and add a setup script or Makefile target that copies them:

    ```bash
    #!/bin/bash
    cp scripts/hooks/post-checkout .git/hooks/post-checkout
    cp scripts/hooks/post-merge .git/hooks/post-merge
    cp scripts/hooks/post-rewrite .git/hooks/post-rewrite
    cp scripts/hooks/pre-commit .git/hooks/pre-commit
    chmod +x .git/hooks/post-checkout .git/hooks/post-merge \
             .git/hooks/post-rewrite .git/hooks/pre-commit
    ```
  </Tab>
  <Tab title="git config core.hooksPath">
    Point git at a tracked directory for hooks across the whole repo:

    ```bash
    git config core.hooksPath scripts/hooks
    ```

    Files in that directory still need to be executable:

    ```bash
    chmod +x scripts/hooks/post-checkout
    ```
  </Tab>
</Tabs>
